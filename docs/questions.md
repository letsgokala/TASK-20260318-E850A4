1. User Account Lifecycle and Credential Storage
Question: The spec requires username/password login with strong hashing and salting but does not define account creation flow, password complexity rules, password reset mechanism, whether admins provision accounts or users self-register, or how initial system administrator seeding works. What is the full account lifecycle, and which hashing algorithm and salt strategy should be used?
Assumption: Accounts are provisioned exclusively by a system administrator — there is no self-registration. The first admin account is seeded via a CLI command at deployment time. Password complexity requires minimum 12 characters, at least one uppercase, one lowercase, one digit, and one special character. No password-reset flow exists; an admin manually resets passwords.
Solution: Create a `users` table with columns `id (UUID PK)`, `username (UNIQUE NOT NULL)`, `password_hash (VARCHAR(255) NOT NULL)`, `role (ENUM: applicant, reviewer, financial_admin, system_admin)`, `is_active (BOOLEAN DEFAULT true)`, `created_at`, `updated_at`, `created_by (FK users.id)`. Use `bcrypt` with a work factor of 12 and a per-user random 16-byte salt (embedded in the bcrypt output). Provide a FastAPI CLI command `python -m app.seed_admin --username <u> --password <p>` that inserts the first `system_admin` row idempotently. Expose `POST /api/v1/admin/users` (system_admin only) for provisioning, and `PUT /api/v1/admin/users/{id}/reset-password` for admin-initiated resets. The frontend admin panel includes a "User Management" page with create/deactivate/reset-password actions. All mutations write to an `audit_logs` table.

2. Account Lockout Mechanics and Clock-Skew Recovery
Question: The spec states 10 failed attempts within 5 minutes triggers a 30-minute lock, but does not define how the sliding window is tracked, whether the counter resets on successful login, what happens if the server clock shifts, or whether an admin can manually unlock an account before the 30 minutes expire.
Assumption: The window is a sliding 5-minute window tracked by storing each failed attempt timestamp. A successful login clears the failure history. Admins can manually unlock any account at any time. The system relies on the local PostgreSQL `now()` function, so clock skew between application and database is irrelevant as long as both run on the same host (which the offline-only deployment guarantees).
Solution: Create a `login_attempts` table: `id (BIGSERIAL PK)`, `user_id (FK)`, `attempted_at (TIMESTAMPTZ DEFAULT now())`, `success (BOOLEAN)`. On each login request, count rows where `user_id = X AND success = false AND attempted_at > now() - interval '5 minutes'`. If count >= 10, reject with HTTP 423 and include a `Retry-After` header. Add `locked_until (TIMESTAMPTZ NULL)` to the `users` table; set it to `now() + interval '30 minutes'` when the threshold is crossed. `POST /api/v1/auth/login` checks `locked_until` first. `PUT /api/v1/admin/users/{id}/unlock` (system_admin only) sets `locked_until = NULL`. A scheduled PostgreSQL function or FastAPI startup task prunes `login_attempts` rows older than 24 hours to prevent table bloat.

3. Role-Permission Model and Isolation Granularity
Question: The spec names four roles (applicant, reviewer, financial_admin, system_admin) but does not define whether permissions are hardcoded per role or stored in a dynamic RBAC table, whether a user can hold multiple roles, whether reviewers are scoped to specific registration categories, or how financial_admin access is isolated from review data.
Assumption: Roles are hardcoded in an enum — no dynamic RBAC. A user holds exactly one role. Reviewers can see all submitted registrations (no category scoping). Financial admins can see registration metadata (title, status, applicant name) for context but cannot access uploaded materials or review comments. System admins have full access.
Solution: Enforce role via the `role` column on `users` as a PostgreSQL `ENUM('applicant','reviewer','financial_admin','system_admin')`. Implement a FastAPI dependency `get_current_user_with_roles(*allowed_roles)` that extracts the JWT claim and raises HTTP 403 if the role is not in the allowed set. Apply this dependency at the router level: `/api/v1/registrations` (applicant, reviewer, system_admin), `/api/v1/reviews` (reviewer, system_admin), `/api/v1/finance` (financial_admin, system_admin), `/api/v1/admin` (system_admin). Row-level isolation: applicants see only their own registrations (`WHERE applicant_id = current_user.id`); reviewers see all with status >= `Submitted`. Create a `permission_matrix` constant in `app/auth/permissions.py` mapping role to list of permitted resource-action pairs for audit and testing purposes.

4. Sensitive-Field Masking Rules per Role
Question: The spec says ID numbers and contact information must be masked "according to role" but does not define which fields are considered sensitive, what the masking pattern is (full redaction, partial mask, hash), or whether masking occurs at the database query level, serialization level, or API response level.
Assumption: Sensitive fields are: `id_number` (national ID), `phone_number`, `email`. Masking is partial: ID numbers show only the last 4 characters (`*********1234`), phone numbers show only the last 4 digits (`*******5678`), email shows the first character and domain (`k***@example.com`). Applicants see their own data unmasked. Reviewers see masked PII. Financial admins see masked PII. System admins see unmasked data.
Solution: Define a Pydantic response model `ApplicantProfile` with a class method `mask_for_role(role)` that returns a copy with masked values. Masking is applied at the serializer layer in FastAPI response models — never at the database level, so audit queries by system admins always access full data. Create utility functions in `app/utils/masking.py`: `mask_id_number(val)`, `mask_phone(val)`, `mask_email(val)`. The `registrations` table stores fields `applicant_id_number (VARCHAR(50))`, `applicant_phone (VARCHAR(30))`, `applicant_email (VARCHAR(255))` encrypted at rest using Fernet symmetric encryption with a key loaded from an environment variable (`SENSITIVE_FIELD_KEY`). On read, decrypt first, then apply the masking function based on the requesting user's role before serialization.

5. Registration Draft Autosave and Multi-Step Wizard State
Question: The spec mentions a form wizard but does not define whether partial/draft registrations are persisted, what triggers a save (explicit save button, timed autosave, page navigation), whether the draft survives a browser refresh or server restart, or how draft data is validated versus final submission data.
Assumption: Drafts are saved server-side. The frontend autosaves every 30 seconds if dirty, and on every wizard step navigation. Drafts persist across browser refresh and server restart. Draft data undergoes only structural validation (correct field types, within length limits) but not business-rule validation (required fields may be empty). Full validation runs only on explicit "Submit" action.
Solution: Add a `status` column to the `registrations` table as `ENUM('draft','submitted','supplemented','approved','rejected','canceled','waitlisted')`. The `registrations` table stores all wizard fields as nullable columns. `PUT /api/v1/registrations/{id}/draft` accepts partial payloads, applies structural validation only, sets `updated_at = now()`, and returns HTTP 200. `POST /api/v1/registrations/{id}/submit` runs full validation (all required fields, numeric ranges, material completeness check) and transitions status from `draft` to `submitted`; returns HTTP 422 with field-level errors on failure. The frontend wizard stores the current step index in `registrations.wizard_step (SMALLINT)` so the user resumes at the correct step on reload. Autosave calls are debounced at 30 seconds on the frontend with a `lastSavedAt` indicator in the UI.

6. Material Upload Storage, Versioning, and Size Enforcement
Question: The spec allows up to 3 versions per material and 200MB total per registration, but does not define whether old versions are retained or overwritten, whether the 200MB cap includes all versions or only the latest, how version numbering works when a version is deleted, or what the directory structure is on the local filesystem.
Assumption: All versions are retained (never overwritten or deleted by the system). The 200MB cap includes the sum of all version files across all materials in the registration. Version numbers are monotonically increasing (1, 2, 3) and are never reused even if a version is logically superseded. Deletion of uploaded files is not permitted — only status transitions.
Solution: Create `materials` table: `id (UUID PK)`, `registration_id (FK)`, `checklist_item_id (FK)`, `created_at`. Create `material_versions` table: `id (UUID PK)`, `material_id (FK)`, `version_number (SMALLINT CHECK 1..3)`, `original_filename`, `mime_type (CHECK IN pdf,jpg,png)`, `file_size_bytes (BIGINT)`, `sha256_hash (CHAR(64))`, `storage_path (VARCHAR(500))`, `status (ENUM: pending_submission, submitted, needs_correction)`, `correction_reason (TEXT NULL)`, `uploaded_at (TIMESTAMPTZ)`, `uploaded_by (FK users.id)`. Unique constraint on `(material_id, version_number)`. Before accepting an upload, query `SELECT COALESCE(SUM(file_size_bytes),0) FROM material_versions mv JOIN materials m ON mv.material_id = m.id WHERE m.registration_id = :rid`. If adding the new file exceeds 200MB (209715200 bytes), reject with HTTP 413. If `version_number` would be 4, reject with HTTP 409. Store files at `/storage/materials/{registration_id}/{material_id}/v{version_number}/{original_filename}`. Endpoint: `POST /api/v1/registrations/{rid}/materials/{mid}/versions` (multipart upload).

7. Deadline Locking and One-Time Supplementary Submission Window
Question: The spec says materials are locked after the submission deadline with a one-time supplementary submission within 72 hours, but does not define where the deadline is stored, who sets it, whether the 72-hour window is automatic or must be explicitly opened by an admin/reviewer, how "one-time" is enforced, or what happens to the registration status during the supplement window.
Assumption: Each registration belongs to a `collection_batch` that has a `submission_deadline (TIMESTAMPTZ)`. Deadlines are set by system admins. The 72-hour supplementary window opens automatically at the deadline for any registration whose status is `submitted` or `needs_correction`. "One-time" means the applicant may upload supplementary materials exactly once within the window — tracked by a boolean flag. The registration status changes to `supplemented` upon supplementary submission.
Solution: Add to `collection_batches` table: `id (UUID PK)`, `name`, `submission_deadline (TIMESTAMPTZ NOT NULL)`, `supplementary_deadline (TIMESTAMPTZ GENERATED ALWAYS AS (submission_deadline + interval '72 hours') STORED)`, `created_by`, `created_at`. Add to `registrations` table: `supplementary_used (BOOLEAN DEFAULT false)`. On any material upload request, the API checks: (a) if `now() <= submission_deadline`, allow normal upload; (b) if `submission_deadline < now() <= supplementary_deadline AND supplementary_used = false`, allow the upload and set `supplementary_used = true`, transition registration status to `supplemented`; (c) otherwise reject with HTTP 403 `{"detail": "Submission window closed"}`. The "one-time" supplementary submission is treated as a single API call where the applicant can upload/update multiple materials in one batch request: `POST /api/v1/registrations/{rid}/supplementary-submit` accepts an array of file uploads. After this endpoint returns 200, `supplementary_used` is set to `true` in the same transaction.

8. SHA-256 Duplicate Detection and Disabled-by-Default Similarity Endpoint
Question: The spec requires SHA-256 fingerprints for duplicate detection and mentions a "duplicate/similarity-check API endpoint that exists but is disabled by default." It does not clarify whether duplicate detection should block uploads, whether it is cross-registration or intra-registration, what "disabled by default" means technically (feature flag, commented-out route, config toggle), or what the similarity check does beyond exact hash matching.
Assumption: SHA-256 duplicate detection operates cross-registration (system-wide). It does not block uploads — it flags them. The disabled endpoint performs exact-hash matching only (no fuzzy/content similarity). "Disabled by default" means controlled by a configuration flag in the application settings.
Solution: On every file upload, compute SHA-256 and store in `material_versions.sha256_hash`. After insert, query `SELECT mv.id, m.registration_id FROM material_versions mv JOIN materials m ON mv.material_id = m.id WHERE mv.sha256_hash = :hash AND mv.id != :current_id`. If matches exist, add a `duplicate_flag (BOOLEAN DEFAULT false)` and `duplicate_of (UUID FK material_versions.id NULL)` to `material_versions` and set them accordingly — but do not reject the upload. Endpoint `GET /api/v1/materials/duplicates?hash={sha256}` returns matching records. This route is registered conditionally: in `app/config.py`, define `ENABLE_DUPLICATE_CHECK_API: bool = False`. In the router, `if settings.ENABLE_DUPLICATE_CHECK_API: router.include_router(duplicate_router)`. Admins toggle via environment variable `ENABLE_DUPLICATE_CHECK_API=true`.

9. Review Workflow State Machine and Transition Rules
Question: The spec lists workflow states (Submitted, Supplemented, Approved, Rejected, Canceled, Promoted from Waitlist) but does not define the valid transitions between states, whether transitions are irreversible, who can trigger each transition, whether "Canceled" can occur from any state, or how "Promoted from Waitlist" is entered (implying a Waitlisted state that is not listed).
Assumption: There is an implicit `Waitlisted` state. Valid transitions: `Submitted -> Approved | Rejected | Waitlisted | Canceled`, `Supplemented -> Approved | Rejected | Waitlisted | Canceled`, `Waitlisted -> Promoted from Waitlist | Canceled`, `Rejected -> (terminal)`, `Approved -> Canceled`, `Promoted from Waitlist -> Canceled`. Only reviewers and system admins may trigger review transitions. Applicants may trigger `Canceled` only from `Submitted` or `Supplemented`. All transitions except to `Canceled` are irreversible.
Solution: Define the state machine in `app/workflows/review_states.py` as a dictionary `VALID_TRANSITIONS: dict[str, list[str]]` mapping each state to allowed next states with required role annotations. Create a `review_records` table: `id (UUID PK)`, `registration_id (FK)`, `from_status`, `to_status`, `comment (TEXT)`, `reviewed_by (FK users.id)`, `reviewed_at (TIMESTAMPTZ)`. Endpoint `POST /api/v1/registrations/{rid}/transition` accepts `{"to_status": "...", "comment": "..."}`, validates against `VALID_TRANSITIONS[current_status]`, checks role authorization, writes the `review_records` row, and updates `registrations.status` — all in a single database transaction. If the transition is invalid, return HTTP 409 with the list of allowed transitions. The frontend review page renders only the buttons for valid next states based on current status and user role.

10. Batch Review Execution and Partial Failure Handling
Question: The spec allows batch review of up to 50 entries but does not define what happens if some entries in the batch fail validation (e.g., an entry is already approved), whether the entire batch is atomic or partial success is allowed, or how results are reported to the reviewer.
Assumption: Batch review is not atomic — partial success is allowed. Each entry is processed independently. The response includes per-entry results so the reviewer can see which succeeded and which failed, and why.
Solution: Endpoint `POST /api/v1/reviews/batch` accepts `{"action": "approve|reject|waitlist", "comment": "...", "registration_ids": [...]}` with a maximum of 50 IDs (validated). The handler iterates each ID, wrapping each in its own `SAVEPOINT`. If the transition is invalid for a given registration (wrong state, permission issue), that savepoint is rolled back, and the error is recorded. The response schema: `{"results": [{"registration_id": "...", "success": true|false, "error": "..."|null}], "succeeded": N, "failed": M}`. HTTP status is 200 if at least one succeeded, 422 if all failed. Every successful transition creates a `review_records` entry. The frontend batch review page shows a result summary modal after submission with green/red indicators per entry.

11. Financial Over-Budget Confirmation and Transaction Integrity
Question: The spec requires a frontend warning when expenses exceed budget by more than 10% and explicit confirmation before saving, but does not define whether the backend enforces this constraint or trusts the frontend, what happens if two concurrent transactions both push the total over budget, or how the budget threshold is stored.
Assumption: The backend independently enforces the over-budget check — frontend confirmation is UX only, not a security boundary. Concurrent transaction safety is ensured via a row-level lock on the funding account during transaction insertion. The budget is stored per funding account.
Solution: Create `funding_accounts` table: `id (UUID PK)`, `registration_id (FK)`, `allocated_budget (NUMERIC(14,2) NOT NULL)`, `created_at`, `updated_at`. Create `financial_transactions` table: `id (UUID PK)`, `funding_account_id (FK)`, `type (ENUM: income, expense)`, `amount (NUMERIC(14,2) NOT NULL CHECK > 0)`, `category (VARCHAR(100))`, `description (TEXT)`, `invoice_attachment_path (VARCHAR(500) NULL)`, `recorded_by (FK users.id)`, `recorded_at (TIMESTAMPTZ)`. Endpoint `POST /api/v1/finance/accounts/{aid}/transactions` accepts an `over_budget_confirmed (BOOLEAN DEFAULT false)` field. Inside the handler: `SELECT ... FROM funding_accounts WHERE id = :aid FOR UPDATE` (row lock), then compute `total_expenses = SUM(amount) WHERE type = 'expense'`. If `(total_expenses + new_amount) > allocated_budget * 1.10 AND NOT over_budget_confirmed`, return HTTP 409 `{"detail": "over_budget", "current_total": ..., "budget": ..., "overage_pct": ...}`. The frontend catches this 409, shows a confirmation dialog, and re-submits with `over_budget_confirmed: true`. The transaction insert and balance check occur within a single serializable transaction to prevent race conditions.

12. Quality Metric Calculation Timing and Alert Thresholds
Question: The spec requires computing approval rate, correction rate, and overspending rate, and triggering local alerts when thresholds are exceeded, but does not define when these metrics are computed (real-time, nightly, on-demand), where thresholds are configured, what "local alerts" are (email, in-app notification, log entry), or at what granularity metrics apply (per batch, per category, system-wide).
Assumption: Metrics are computed on-demand when the dashboard is loaded and cached for 5 minutes. Thresholds are stored in a configuration table and editable by system admins. "Local alerts" are in-app notifications (no email in an offline system). Metrics are computed per collection batch and system-wide.
Solution: Create `alert_thresholds` table: `id (SERIAL PK)`, `metric_name (VARCHAR(50) UNIQUE)`, `threshold_value (NUMERIC(5,2))`, `comparison (ENUM: gt, lt)`, `updated_by (FK)`, `updated_at`. Seed defaults: `approval_rate lt 50.0`, `correction_rate gt 30.0`, `overspending_rate gt 15.0`. Endpoint `GET /api/v1/metrics?batch_id={optional}` computes: `approval_rate = approved / total * 100`, `correction_rate = needs_correction / total * 100`, `overspending_rate = over_budget_transactions / total_transactions * 100`. Results are cached in-memory (FastAPI `lru_cache` with TTL via `cachetools`). After computation, compare each metric against `alert_thresholds`; if exceeded, insert into `notifications` table: `id`, `user_id (NULL = all admins)`, `message`, `severity (warning|critical)`, `read (BOOLEAN DEFAULT false)`, `created_at`. The frontend dashboard header shows a bell icon with unread notification count, polled every 60 seconds via `GET /api/v1/notifications?unread=true`.

13. Daily Local Backup Scope and One-Click Recovery Procedure

Assumption: Backup covers both the PostgreSQL database and the `/storage/materials/` filesystem. Retention is 30 days (rolling). Backups are stored on a separate local disk or partition at `/backups/`. One-click recovery means: admin selects a backup date from a list and confirms; the system restores both the database and files from that snapshot. The application enters maintenance mode during restore.
Solution: Create a bash script `scripts/backup.sh` that runs `pg_dump --format=custom` to `/backups/db/backup_YYYYMMDD.dump` and `rsync -a /storage/materials/ /backups/files/YYYYMMDD/`. Schedule via `cron` at 02:00 daily. A cleanup step removes backups older than 30 days. Expose `GET /api/v1/admin/backups` (system_admin only) returning available backup dates by listing the `/backups/db/` directory. `POST /api/v1/admin/backups/{date}/restore` sets a `maintenance_mode` flag in a file `/tmp/app_maintenance`, which a FastAPI middleware checks — returning HTTP 503 for all non-admin endpoints. The restore process runs `pg_restore --clean --if-exists` and `rsync -a --delete /backups/files/{date}/ /storage/materials/`. On completion, the maintenance flag is removed. The frontend admin page shows a backup list with a "Restore" button and a confirmation dialog warning that current data will be overwritten.

18. Sensitive Configuration Encryption and Secret Management
Question: The spec requires "encryption of sensitive configuration values" but does not define which configuration values are considered sensitive, what encryption mechanism is used, where the encryption key is stored (given the offline-only constraint rules out cloud KMS), or how key rotation is handled.
Assumption: Sensitive configuration values are: `DATABASE_URL` (contains credentials), `SECRET_KEY` (JWT signing), `SENSITIVE_FIELD_KEY` (PII field encryption). In an offline environment, these are stored as environment variables in a `.env` file that is encrypted at rest using `age` (a modern file encryption tool) with a passphrase known only to the system administrator. The application reads plaintext environment variables at runtime (injected by Docker from the decrypted `.env`).
Solution: The `.env` file contains plaintext secrets during runtime but is stored encrypted on disk as `.env.age` when not in use. Provide a helper script `scripts/encrypt_env.sh` that runs `age -p -o .env.age .env && shred -u .env` and `scripts/decrypt_env.sh` that runs `age -d -o .env .env.age`. The `docker-compose.yml` references `env_file: .env`, which must be decrypted before running `docker-compose up`. Document this in the deployment guide. For key rotation of `SENSITIVE_FIELD_KEY`: provide a CLI command `python -m app.rotate_key --old-key <old> --new-key <new>` that reads all encrypted PII fields, decrypts with the old key, re-encrypts with the new key, and updates rows in a transaction. `SECRET_KEY` rotation invalidates all active JWTs (acceptable — users simply re-login). Add a `KEY_VERSION (SMALLINT DEFAULT 1)` column to tables with encrypted fields to support concurrent old/new key decryption during rolling rotation.
